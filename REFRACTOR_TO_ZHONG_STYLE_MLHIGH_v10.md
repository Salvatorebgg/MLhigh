# MLhigh v10：纯小白变量选择驱动的后端校验工作流

## 核心目标

用户只需要在“选择变量”中把变量分到三类：
- 研究变量
- 协变量 / 混杂因素
- 结局变量

系统不要求用户理解每种统计方法需要什么参数，而是由后端根据真实数据类型、变量角色和每个方法的代码运行条件，判断哪些方法能做、哪些方法不能做。

## 本轮关键修改

1. 新增后端 `/api/validate-methods`
   - 前端变量选择变化后，会把三类变量角色传给后端；
   - 后端基于真实 DataFrame 判断变量类型；
   - 后端为每个方法自动推导运行参数；
   - 返回每个方法是否可运行、不可运行原因、自动推导参数。

2. 前端方法可用性不再靠猜
   - 方法列表优先显示后端校验结果；
   - 后端判断不可运行的方法自动变灰；
   - 后端校验通过的方法才允许选择；
   - 运行分析时再次携带 role_vars 给后端做二次推导和校验。

3. 连续结局和分类结局严格区分
   - 分类模型只允许二分类/多分类离散结局；
   - 连续结局会让分类模型变灰；
   - 连续结局可用于 Lasso/岭回归或相应高级统计模型。

4. 运行兜底保护
   - 后端运行前再次检查目标变量与方法兼容性；
   - ValueError 和其他异常转成 400 友好错误，不再 Internal Server Error；
   - 可运行方法尽量保证能跑到底，不可运行就前置变灰。

5. 提示词样式调整
   - “综合示例数据已加载，请先查看数据预览，再进入变量选择”等提示字体增大；
   - 提示放到右侧工作区左下方；
   - 删除左侧蓝色/绿色竖条小框。

## 已验证

- 前端 JS 语法检查通过；
- 后端 Python 编译通过；
- `/api/validate-methods` 测试通过：
  - outcome 作为分类结局时，ml_lr 可运行；
  - outcome_continuous 作为连续结局时，ml_lr 变为不可运行；
  - outcome_continuous 时，ml_lasso 可运行；
- `/api/analyze` 使用 role_vars 运行 ml_lr 可返回 200。
